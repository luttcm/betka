# AGENTS: продукт и правила реализации
UPD: Каждый раз при новой задаче создавай в гите ветку от мастера

## 1) Контекст продукта

Букмейкерская платформа для пользовательских событий.

- Пользователь создаёт событие (пример: «Закроется ли компания X в 2026 году?»).
- Событие проходит модерацию.
- После одобрения другие пользователи делают ставки на исходы.
- Платформа рассчитывает коэффициенты и удерживает комиссию (маржу).

## 2) Ключевое решение по репозиториям

Для текущего этапа выбрать **monorepo** (frontend + backend + docs + infra в одной репе).

Причины:

1. Быстрее синхронизировать контракт API и клиент.
2. Проще единый CI/CD, качество и versioning на MVP.
3. Проще поднимать окружение одной командой через Docker Compose.
4. Меньше операционной сложности до выхода на стабильный продуктовый ритм.

Когда переходить к split-repo:

- разные команды с независимыми релизными циклами,
- заметно разные требования по безопасности/доступам,
- отдельные пайплайны и SLA по сервисам.

## 3) Базовый стек (текущий)

- Backend: Go (Gin)
- DB: PostgreSQL
- Cache/Queue: Redis
- Infra: Docker, Docker Compose
- CI: GitHub Actions
- Docs: Markdown

## 4) План работ (архитектура, БД, тесты, документация, CI/CD, лицензия, Docker)

### Этап A — Repo и инфраструктурный baseline

1. Зафиксировать monorepo-структуру (`backend/`, `frontend/`, `docs/`, `plans/`, `.github/`).
2. Довести Docker Compose для локального запуска api/worker/postgres/redis.
3. Согласовать переменные окружения и `.env.example`.

### Этап B — Домен и БД

1. Описать доменную модель: users, wallets, wallet_transactions, events, event_outcomes, odds_snapshots, bets, moderation_tasks, audit_logs.
2. Подготовить миграции PostgreSQL и ограничения целостности.
3. Зафиксировать индексы под ключевые сценарии чтения/записи.

### Этап C — Backend API

1. Auth + RBAC. Надо еще сделать так, чтоб при регистрации пользователя ему на почту прилетало сообщение о регистрации и подтверждении
2. Поток: создание события → модерация → публикация.
3. Поток ставок: проверка, холд, идемпотентность.
4. Settlement и проводки кошелька.

### Этап D — Frontend

1. Каталог событий, карточка события.
2. Создание события.
3. Кошелёк и мои ставки.
4. Модерация для ролей moderator/admin.

### Этап E — Качество и безопасность

1. Unit/integration/e2e тесты.
2. Логи, health/readiness.
3. Базовые security-практики (RBAC, rate limit, аудит).

### Этап F — Документация, лицензия, релизы

1. Актуализировать README и docs/*.
2. Поддерживать CONTRIBUTING и LICENSE.
3. Укрепить CI/CD quality gates и сборку Docker image.

## 5) Definition of Done для MVP

1. Пользовательский e2e-сценарий проходит полностью: регистрация → создание события → модерация → ставка → settlement.
2. Финансовые операции выполняются транзакционно и воспроизводимо.
3. Документация и CI/CD позволяют поднять проект «с нуля» без ручных шагов.
